# Ziply Plan

## Product Rule

Ziply only claims formats that it handles itself.

That means:

- no calling `unar`
- no calling `7z`
- no calling `7zz`
- no calling `unrar`
- no hidden bridge to another archive app or system package

If a format is not implemented natively inside Ziply, it must stay out of active support and out of shell/file-association promises.

## Scope Clarification

`All formats` cannot be treated as “every archive format ever invented”.

For Ziply, the practical rule is:

1. Every format listed in the app UI must be implemented natively in Ziply.
2. Every format listed in README must be backed by native code inside this repository.
3. New formats are only promoted from `planned` to `supported` after native create/extract coverage, tests, and UX support are complete.

## Native-Only Milestones

### 1. Keep the current native core stable

Status: in progress

- Keep `zip`, `tar`, `tar.gz`, `tar.bz2`, `tar.xz`, `xz`, `bz2`, `gz`, and `7z` reliable across macOS, Windows, and Linux
- Preserve current capabilities:
  - preview
  - selective extract where implemented
  - queue
  - history
  - live jobs
  - conflict handling
  - password flows already shipped
- Add regression coverage whenever archive behavior changes

### 2. Finish native quality for current formats

Status: in progress

- Add more fixture-driven tests for:
  - nested directories
  - unicode filenames
  - large files
  - empty files
  - empty directories
  - conflict handling edge cases
  - password failures
- Add archive compatibility validation against files produced by common tools
- Add cross-platform CI verification for Windows, macOS, and Linux bundle builds

Completed in this track so far:

- compatibility fixtures for `zip` from `/usr/bin/zip` and `/usr/bin/ditto`
- encrypted `zip` fixture coverage with password success and failure cases
- compatibility fixtures for `tar`, `tar.gz`, `tar.bz2`, and `tar.xz`
- compatibility fixtures for raw `gz`, `bz2`, and `xz` streams
- compatibility fixtures for `7z`, including encrypted `7z`, generated by `7zz`
- CI smoke-build verification added for macOS, Windows, and Linux with `tauri build --no-bundle`
- release workflow now verifies installer outputs and downloaded release assets before upload and publish
- CI and release workflows now fail early on version mismatch across package, Tauri, and Cargo manifests

### 3. Native encryption parity

Status: in progress

- Expand password UX:
  - stronger error reporting
  - clearer unsupported-format messaging
  - fixture coverage for encrypted archives

Completed in this track so far:

- native password-protected ZIP creation shipped
- encrypted ZIP extraction coverage kept in place
- encrypted ZIP and encrypted 7Z compatibility fixtures added

### 4. Expand native format coverage carefully

Status: planned

Candidate formats to evaluate in order:

1. split or multipart archives where native libraries are good enough
2. `rar` only if native implementation quality and licensing are acceptable

Promotion rule for any new format:

- native parser or native writer exists inside Ziply’s code or directly linked library stack
- no runtime dependency on another archive app
- create and extract behavior is testable
- preview support is defined
- shell/file association changes happen only after backend support is real

### 5. RAR-native investigation track

Status: research only

`RAR` is the hardest item in this roadmap.

Requirements before `RAR` can be marked supported:

- no dependence on external CLI tools or installed archive apps
- acceptable legal and licensing posture
- support quality good enough for production use
- native extraction at minimum
- preview and selective extract strategy defined

Decision gate:

- if native `RAR` is not technically or legally clean enough, Ziply should keep `RAR` unsupported instead of shipping a fake support story

## Definition Of Done For Native Support

A format is `supported` only when all of the following are true:

- Ziply can open it natively
- Ziply can extract it natively
- Ziply can create it natively if creation is claimed
- Ziply has automated tests for it
- Ziply documentation matches the real behavior
- Ziply UI does not mention external helper apps for that format

## Immediate Implementation Track

1. Keep the codebase native-only and remove any future archive bridge shortcuts.
2. Harden the current native formats with more compatibility and regression tests.
3. Add the next native formats one by one, only after backend quality is real.
4. Treat `RAR` as research until native support is proven, not assumed.
